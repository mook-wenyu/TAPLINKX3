# 产品策略

## 1. 原始问题

在 AR 眼镜场景里，用户缺少一个低负担、低误触、可跨应用复用的输入通道。

要解决的问题不是“做一个炫酷的手势系统”，而是：

- 如何让用户在不依赖触控板的前提下完成选择与确认。
- 如何让跨应用控制具备稳定性、可撤销性和可预测性。
- 如何在普通第三方 App 权限边界内，交付一个真实可用的系统能力。

## 2. 产品定义

短期产品不定义为“空中鼠标”，而定义为：

`AR 眼镜上的跨应用语义控制层`

它把用户的低负担手部意图，映射成 Android 已支持的语义动作：

- 下一项
- 上一项
- 激活当前焦点
- 返回
- 主页
- 条件滚动

## 3. MVP 边界

### 做什么

- `Accessibility-first`
- `pinch confirm`
- `focus navigation`
- 极简 HUD 焦点反馈
- 仅支持结构良好的可访问节点树
- 首页只提供一个围绕无障碍服务的主引导目标

### 不做什么

- 全局空气鼠标
- 任意像素点击
- 拖拽与复杂画布交互
- 复杂手势语法
- 未经确认的厂商私有深度接口依赖
- 在主入口提前暴露悬浮窗总开关

## 4. 长期演进

### 阶段 1：MVP

- 通用语义焦点导航
- 捏合确认
- 基础全局动作

### 阶段 2：全局语义控制

- 跨应用统一动作模型
- 更好的失败恢复
- 更可靠的滚动/分组导航

### 阶段 3：白名单深适配

- 高价值 App 的节点分组与任务流优化
- 应用级 landmark 与快捷语义动作
- 更细粒度的容错与回退

## 5. 产品原则

- `KISS`：焦点先行，确认明确。
- `YAGNI`：不为未来可能存在的厂商接口提前设计复杂框架。
- `SOLID`：手势识别、语义执行、HUD 展示三层解耦，采用单模块下的包级分层而不是过早多模块化。
- `可测试`：状态机、动作映射、App 支持等级都应独立验证。
- `低耦合`：手势引擎可替换，语义层不绑定特定 CV 栈。
- `AR-first onboarding`：首页优先解决“核心能力是否启用”，而不是堆叠系统能力控制台。

## 6. 首页入口决策

当前首页不应设计成通用 Android 设置页，也不应做成双开关控制面板。

推荐决策：

- **保留一个主入口对象**：`语义控制服务`
- **展示形式**：状态卡片或主 CTA，而不是伪装成应用内即时切换的系统开关
- **状态语义**：`未启用 / 已启用 / 需要检查`
- **主动作**：引导用户进入系统无障碍设置完成启用或校验

当前不推荐加入第二个主开关 `悬浮窗`，原因如下：

- 当前项目尚未申请 overlay 权限
- `FocusHudOverlayController` 仍是占位实现，不是成熟功能
- AR 眼镜首页应一屏一目标，避免把未成熟能力包装成主功能
- 在功能、权限、失败恢复尚未完整前，过早暴露 overlay 只会增加理解成本和维护负担

只有在以下条件同时满足后，才重新评估是否加入 overlay 次级控制：

1. overlay 变成真实可交付能力，而不是日志占位
2. 权限申请、权限检测、失败处理与回流路径全部具备
3. 它对核心任务完成率有明确增益
4. 它不会与无障碍主路径竞争首页主注意力

## 7. 成功标准

MVP 的成功不是“识别了多少种手势”，而是：

- 用户是否能稳定完成核心任务。
- 用户是否知道当前焦点是什么。
- 用户是否知道动作会不会发生。
- 用户是否能在误判时立刻撤销。
